- Q-02 (асимметричный guard в одном диффе): тот же аудит — `cmd_price` создаёт `ThreadPoolExecutor(max_workers=len(regions))` без верхнего предела, хотя тем же диффом на ~200 строк ниже введён `_BROWSE_COUNT_CAP` — guard от fan-out для другого пути; симметричный guard для regions не поставлен (источник: dmitry-programmer, ревью диффа). hits=1 → ждём повтор того же механизма (guard введён для одного fan-out пути в диффе, симметричный путь того же класса оставлен голым) в другой ситуации → тогда candidate.
- Q-03 (коммерческие данные владельца из захваченного partner-API ответа перенесены в публичный артефакт без зачистки): свежесобранный ответ `IPartnerFinancialsService.GetAppWishlistReporting` (реальный appid игры владельца + реальная дата + реальные wishlist adds/deletes/purchases) использован как основа тестовой фикстуры `tests/fixtures/wishlist_reporting.json` — при этом в том же файле уже стоял корректно сформулированный комментарий о недопустимости реальных данных в публичных фикстурах, но он не остановил копирование структуры ответа вместе с данными; те же числа продублированы в описание задачи `.boardown/` (источник: code-reviewer, ревью диффа до коммита). hits=1, санкции владельца нет (нашёл суб-агент на ревью, не владелец лично) → уже стоит частичный структурный гард — regression-тест `test_fixture_carries_no_real_owner_data` (падает на реальном appid/дате именно этой фикстуры), но он узкий (одна фикстура, не правило) → ждём повтор того же механизма (живой ответ partner-scoped эндпоинта используется как основа любого публичного артефакта — новой фикстуры, доски, доков — без зачистки коммерческих значений) в другой ситуации → тогда candidate.
